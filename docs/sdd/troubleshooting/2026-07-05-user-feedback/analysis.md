## 報告事項と根本原因

検証楽譜は「**1パート（P1）・2段譜（staves=2、音符に`<staff>1|2</staff>`）**」という標準的なMuseScoreピアノ譜。現アプリは「右手=1パート目、左手=2パート目」という**パート単位の手判定**を前提としており、複数の報告の根が共通している。3系統の並列調査で以下を特定した。

### 原因群A: パーサが `<staff>` を無視（手の判定がパート単位）

- `parser.ts` は `<staff>` 要素を読まず、`Note` 型にもstaffフィールドがない。単一パート譜では clef 配列の先頭（=上段のG）だけを見て `Part.hand='right'` になる（`hand-detector.ts:21-23`）
- 影響: **「運指提案で左手にすると対象パートがない」**（`FingeringPanel/index.tsx:36-41` のフィルタが空）、右手運指も実は両手全音を単旋律としてDPに投入していて不正、練習対象の左手モードが機能せず右手モードは全音要求、鍵盤の左右色分け全滅、グレーアウトも段単位にできない
- **tick計算と両手同時グループ化は既に正しい**（`<backup>`処理をこの楽譜で机上検証済み。636グループ中283が両段同時刻を正しく合流）。壊れているのはhandの付与と消費のみ
- 対応: Note単位の `staff`/`hand` を導入（staves≥2のパートは staff1=right / staff2=left、複数パート時は従来のパート判定をフォールバック）。noteId採番・アノテーション互換に影響なし

### 原因群B: 楽譜座標マッピングの陳腐化と採番不整合

- **リサイズずれ**: `autoResize: true` によりOSMDがリサイズで自動再レイアウトするが、クリック解決用の `noteIdToSvgCoord`（`buildNoteIdMap`、load時に1回だけ構築）を再構築しないstaleキャッシュが直接原因。オーバーレイ（運指番号・ハイライト・ループ枠）もリサイズで消える。**setZoom経路にも同種の欠陥**（render後にbuildNoteIdMap未実行）
- **noteId採番の不整合（調査で発見した潜在バグ）**: パーサはXML文書順（staff1全音→backup→staff2）で採番、OSMD側（`osmd-controller.ts:552-562`）はタイムスタンプ順で採番。多声部・2段譜の小節では**同じnoteIdが別の音を指し**、運指表示・ハイライト・右クリック解決が誤った音に紐付く
- 対応: `autoResize:false`＋ResizeObserverで「render→マップ再構築→オーバーレイ再適用」を自前制御（dispose追加）。noteIdマップは独立採番をやめ、**パース済みScoreと（小節番号・タイムスタンプtick・midiNumber・staff）で照合**して構築する

### 原因群C: 運指提案の和音処理

- **「和音でも1音分しか運指されない」の直接原因は表示**: 計算・保存は全構成音に行われているが、`buildNoteIdMap` が和音全音に同一座標を登録するため、指番号が同一位置にスタックして1個に見える（`osmd-controller.ts:555-559`、`renderFingeringLayer`）
- **DP自体も和音非対応**: 同一tickの構成音を「順に弾く音」として遷移コストを課すため、音高順と指順の整合・和音内スパン制約がなく物理的に不可能な運指になりうる
- 対応: 同一startTickをコードユニット化するDP拡張（右手=音高昇順に指昇順のコンビネーション C(5,k)、SPAN_TABLEでユニット内実行可能性チェック、ユニット間は既存遷移コスト。worker/型プロトコル変更不要）＋符頭単位の描画座標

### 原因群D: UI・機能不足（単純）

- **チェックボックスのラベルが読めない**: electron-viteテンプレート残骸の `base.css` が `--color-text: rgba(255,255,245,0.86)`（ダークテーマ用のほぼ白）をbodyに設定しており、色指定のない「メトロノーム」「ループ」ラベルが継承して薄グレー化。ツールチップは既に存在する。対応: ラベルの明示色＋base.cssのテンプレ残骸是正
- **音量変更不可**: マスターボリューム制御が存在しない。対応: Tone.js Destinationの音量制御＋ツールバーUI＋設定永続化
- **カーソル位置から再生されない**: 再生は常に先頭から。対応: Transport開始オフセット=現在の判定グループのstartTick
- **小節頭単位でしかカーソル移動できない**: `handleNoteClick` が `resetToMeasure` を呼ぶため。対応: クリックで解決したnoteIdの判定グループへ直接移動する `resetToPosition` の導入
- **練習対象と再生パートが無関係**: `loadScore` は常に全パート。対応: Note.hand導入後、practiceModeに応じた再生フィルタ（左手練習=左手のみ再生、右手=右手のみ、両手=全パート）※US-010のスコープ更新
- **D&Dで開けない**: ドロップハンドラ不在。対応: ドロップで既存のオープン経路へ結線（メインプロセスの書き込みallowlist登録を含む）
