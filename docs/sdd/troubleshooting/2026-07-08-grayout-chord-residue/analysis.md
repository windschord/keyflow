# 練習対象切替で両段がグレーアウトされる問題の分析（2026-07-08）

## 問題事象

| 項目 | 内容 |
|------|------|
| 現象 | 練習対象を左手/右手に切り替えると、楽譜上の音符が両方の段ともグレーアウトされる。鍵盤ガイドは正しい |
| 期待動作 | 選択した手以外のノートのみ減光される（REQ-002-007） |
| 再現手順 | 和音を含むピアノ譜を開く → 練習対象を両手→左手→右手…と切り替える |
| 発生環境 | ユーザー実機（MuseScore由来のピアノ譜）。実バイナリ+Playwrightで決定的に再現済み |
| 報告 | 2026-07-08 ユーザー実機確認（スクリーンショットあり） |

## 根本原因

**グレーアウトの元opacity復元ロジック（TASK-060導入）が、和音のように複数のnoteIdが同一のSVG要素を共有するケースで破綻する。** Phase 15とは無関係の既存バグ（mainにも存在）。

該当コード（`src/renderer/src/components/ScoreRenderer/osmd-controller.ts`）:

- `grayoutAppliedElements`（146-149行）が `Map<noteId, {element, originalOpacity}>` と**noteId単位**で管理
- `renderGrayoutLayer()`（343-368行）が noteIdごとに `element.style.opacity` を「元値」として読み取ってから0.5を設定
- `restoreGrayoutOpacity()`（371-380行）が記録した元値へ戻す

### 破綻のメカニズム

和音（例: noteId N2, N3）はOSMD/VexFlow上で同一のStaveNote SVG要素を共有する。

1. N2処理: 元opacity=''を記録 → 0.5を設定
2. N3処理: **同じ要素**の opacity は既に'0.5' → 元値として'0.5'を誤記録
3. 復元時: N2で''へ戻す → 直後にN3が'0.5'を再適用 → **減光が残留**

`setGrayedOutNotes` は切替のたびに復元→再適用するため、モード切替を繰り返すと過去に減光した和音が累積残留し、両段がグレーに見える。和音のない譜面では発現しない（既存テストが見逃した理由）。

## 各仮説の判定

| 仮説 | 判定 | 根拠 |
|------|------|------|
| グレー対象集合の算出（hand判定・フィルタ） | 棄却 | Note.hand/staffは正しく、集合も正しい（診断計装で確認） |
| noteId→SVGマッピング不整合 | 棄却 | 全noteIdが正しい要素に対応（ダンプで確認） |
| Phase 15の回帰 | 棄却 | osmd-controller.ts / ScoreRenderer はmainと差分ゼロ |
| main既存バグ | **支持** | 上記メカニズム。和音を含む譜面で顕在化 |

## 再現結果（実バイナリ + Playwright、和音入り1パート2段フィクスチャ）

- 初期(both): 減光0 → 左手: 6（正しい） → **bothへ戻す: 1（0のはず）** → 右手: 5 → **bothへ戻す: 2（累積）**
- 和音を含まないフィクスチャでは正常

## 修正方針の選択肢

| 案 | 内容 | 評価 |
|----|------|------|
| A) 要素単位の管理へ変更（推奨） | `grayoutAppliedElements` をSVG要素単位に持ち替え、対象要素を重複排除してから元opacityを1回だけ記録・復元する | 元値の保全と和音対応を両立。変更はosmd-controller内に閉じる |
| B) 二重記録ガード | 記録済み要素はoriginalOpacityの再取得をスキップ | 最小差分だがnoteId単位管理の歪みが残る |
| C) 元値保存を廃止しremovePropertyで復元 | 構造は最簡だが、OSMDが別目的で設定したopacityを破壊し得る | リスクあり |

いずれの案でも、再発防止として「和音（複数noteIdが同一SVG要素を共有）を含む譜面でモード切替後に減光が完全解除される」ユニットテストを対で追加する。
