# US-009: 運指提案

## ユーザーストーリー

**私は** ピアノ練習者として
**〜したい** AIが最適な運指を提案してくれる機能が欲しい
**なぜなら** 特に初心者・中級者は適切な運指を知ることで効率的に練習できるから

## 受入基準（EARS記法）

### 機能要件

- **REQ-009-001**: ユーザーが「運指提案」を実行した時、システムはTypeScriptで実装された運指エンジンを使って曲全体の運指番号を計算しなければならない
- **REQ-009-002**: 運指提案は右手・左手それぞれに対して独立して実行しなければならない
- **REQ-009-003**: 運指提案の計算中、システムは進捗状況をプログレスバーで表示しなければならない
- **REQ-009-004**: 提案された運指番号が計算完了した時、システムはUS-008と同じ形式で楽譜上に自動表示しなければならない
- **REQ-009-005**: ユーザーが提案された運指を「承認」した時、システムはその運指をアノテーションとして保存しなければならない
- **REQ-009-006**: ユーザーが提案された運指番号を個別に上書き修正できなければならない
- **REQ-009-007**: 運指エンジンはUIスレッドをブロックしないよう、Web Worker上で非同期実行しなければならない

## 運指アルゴリズム要件

運指提案エンジンはPython/music21と同等の品質を達成するため、以下のアルゴリズムを独自実装（TypeScript）する：

- **REQ-009-A01**: エンジンはParncutt-Terzuoloモデルに基づく動的計画法（DP）で最適運指列を計算しなければならない
- **REQ-009-A02**: エンジンは指ごとの手のスパン制約（隣接指の到達距離上限）を考慮しなければならない
- **REQ-009-A03**: エンジンは親指くぐりと指越えの遷移コストを評価しなければならない
- **REQ-009-A04**: エンジンは黒鍵への4・5指の使用を避けるよう重み付けしなければならない
- **REQ-009-A05**: ユーザーが手の大きさ（オクターブ到達限界）を設定した時、システムはそれをスパン制約に反映しなければならない
- **REQ-009-A06**: エンジンはスケール・アルペジオの定型運指パターンを優先的に適用しなければならない

## 画面/UI要件

- 「運指提案」ボタンをツールバーに配置する
- 提案結果は楽譜上に薄い色（未承認）で表示し、承認後に濃い色（確定）に変わる
- 提案前・提案後の比較表示が可能

## データ要件

- 実装言語: TypeScript（Electron Renderer プロセスの Web Worker）
- 外部ランタイム依存: **なし**（Python・Node.js のインストール不要）
- 計算結果はUS-008と同じアノテーション形式（JSON）で保存
