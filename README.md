# keyflow — MusicXML Piano Practice App

English | [日本語](./README.ja.md)

A desktop app for practicing piano with your own MusicXML scores and a MIDI keyboard.
It aims to deliver a Synthesia / Piano Marvel-like experience with the sheet music you already own.

> **Status**: Pre-release — v0.1.0 release candidates are available

---

## Features

| Feature | Description |
|---------|-------------|
| 🎼 MusicXML import | Load `.xml` / `.musicxml` / `.mxl` files and render them as standard notation |
| 🤚 Hands-separate practice | Switch between right-hand only, left-hand only, and both-hands modes |
| 🎹 MIDI note checking | Match MIDI keyboard input against the score and judge it in real time |
| 💡 Key guide | Visually highlight the next keys to press on the on-screen keyboard |
| ⏱️ Tempo control | Practice at 20%–200% of the original tempo |
| 🔁 A-B loop | Select a measure range and repeat it until it sticks |
| ✍️ Fingering annotations | Write finger numbers and comments on notes and save them |
| 🤖 Fingering suggestions | Automatic fingering powered by a Parncutt-Terzuolo dynamic-programming model |

## Verifying Release Artifacts

Artifacts attached to GitHub Releases (.exe / .dmg / .zip) can be verified in two ways.
Note that code signing / notarization is not yet supported, so the OS shows a warning on first launch (known limitation).

### SHA256 checksums

Every release includes a `SHA256SUMS.txt`. Download it into the same folder as the artifacts and verify:

```bash
# Linux
sha256sum -c SHA256SUMS.txt

# macOS (use shasum since sha256sum is unavailable)
shasum -a 256 -c SHA256SUMS.txt
```

```powershell
# Windows (PowerShell): compute the hash and compare it against SHA256SUMS.txt manually
Get-FileHash .\keyflow.Setup.<version>.exe -Algorithm SHA256
Get-Content .\SHA256SUMS.txt
```

### Build provenance attestation

Artifacts carry a provenance attestation generated by GitHub Actions.
You can verify which workflow and commit produced them:

```bash
gh attestation verify <downloaded-file> --repo windschord/keyflow
```

## Development

### Prerequisites

- Node.js 20.19+ or 22.12+ (developers only — end users do not need it)
- npm 9+

### Setup

```bash
git clone https://github.com/windschord/keyflow.git
cd keyflow
npm install          # postinstall downloads the Electron binary
npm run dev          # start the dev server
```

### Common commands

```bash
npm run dev           # development mode (hot reload)
npm run build         # production build
npm run build:win     # build the Windows NSIS installer
npm run build:mac     # build macOS packages (dmg/zip, arm64+x64)
npm run test          # unit tests
npm run test:coverage # coverage report
npm run test:e2e      # end-to-end tests (Playwright for Electron; runs `npm run build` first)
npm run lint          # lint
npm run typecheck     # TypeScript type check
```

### Appearance in development mode (US-011, TASK-080)

When launched with `npm run dev`, parts of the official app name and icon fall back to
Electron's defaults. The Dock icon (macOS) is applied even in development mode via
`app.dock.setIcon()` (piano + eighth note), but the application menu name in the macOS
menu bar comes from the development Electron binary and cannot be changed (a known
Electron limitation). Check the final appearance (app name and all icons) with a
packaged build such as `npm run build:mac`.

### Pre-release checks (TASK-084)

Before a macOS release, run `npm run build:mac` to create a packaged build, then run the
launch smoke test `npm run test:packaged` and confirm the main window appears.

### Release procedure (for maintainers)

Releases are automated by pushing a tag (`.github/workflows/release.yml`):

1. **Preflight**: main is green (CI passing) and the `npm run build:mac` → `npm run test:packaged` smoke test passes (see "Pre-release checks" above).
2. **Tag and push** (the `v` prefix is required — it triggers the workflow):

   ```bash
   git checkout main && git pull
   git tag v0.1.0
   git push origin v0.1.0
   ```

3. **Watch the workflow**: three jobs run — `build-windows` / `build-macos` / `release`. The two build jobs attach build provenance attestations to the artifacts, and the `release` job uploads all artifacts plus `SHA256SUMS.txt` to a single GitHub Release.

   ```bash
   gh run watch                        # follow the running jobs
   gh run list --workflow=release.yml  # check the status of all three jobs
   ```

4. **Verify the artifacts**: confirm everything is attached (Windows .exe, macOS .dmg×2 / .zip×2, `SHA256SUMS.txt`), then verify attestations and checksums.

   ```bash
   gh release download v0.1.0 -D ./verify   # fetch all artifacts and SHA256SUMS.txt
   cd verify
   for f in *.exe *.dmg *.zip; do            # verify provenance for every artifact (.exe/.dmg/.zip)
     gh attestation verify "$f" --repo windschord/keyflow
   done
   shasum -a 256 -c SHA256SUMS.txt          # on Linux: sha256sum -c
   ```

Code signing / notarization is not supported yet (`identity: null` in `electron-builder.yml`); the OS warning on first launch is a known limitation.
To publish a test pre-release, use a SemVer pre-release tag with a hyphen such as `v0.1.0-rc.1`.
The workflow automatically marks hyphenated tags as pre-releases, so they never surface as the regular latest release.

### About the fingering engine

The app ships a custom fingering engine implemented **entirely in TypeScript** — no
Python/music21 dependency. Based on the dynamic-programming model of
[Parncutt & Terzuolo (1997)](https://doi.org/10.2307/40285402), it computes optimal
fingerings using eight cost functions including span constraints, thumb passing, and
weak-finger costs. It runs asynchronously in a Web Worker, so the UI stays at 60fps
during computation.

### About the bundled grand piano samples

The grand piano voice (the default playback voice) bundles a reduced sample set from
**Salamander Grand Piano V3**
(by [Alexander Holm](https://archive.org/details/SalamanderGrandPianoV3), CC-BY 3.0).

- Attribution: **Salamander Grand Piano V3 by Alexander Holm (CC-BY 3.0)**
- Bundled files: `src/renderer/src/assets/samples/salamander/*.mp3`
  (minor-third spacing A/C/D#/F# per octave, 30 files from A0 to C8, a single velocity
  layer, ~2MB total)
- Source: [the converted mp3 files distributed by the Tone.js project](https://tonejs.github.io/audio/salamander/)
  (the original Internet Archive release converted and hosted by Tone.js; downloaded once
  at development time and bundled, so the built app works fully offline)
