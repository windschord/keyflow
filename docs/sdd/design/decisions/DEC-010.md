# DEC-010: 楽譜ライブラリの永続化をMain側の独立electron-storeで行う

## ステータス

承認済（2026-07-12）

## 文脈

US-017（楽譜ライブラリ）では、取り込んだ楽譜のメタデータ（パス・タイトル・作曲者・
登録日時・最終利用日時）を永続化し、一覧・検索・並べ替え・削除できる必要がある。
永続化方式と管理プロセスの選択肢を比較する。

## 決定

**Mainプロセスの `LibraryService` がライブラリを永続化する。**
**保存先はアプリ設定とは別のelectron-storeインスタンスとし、ストア名は `library` とする。**

- ストアは `{ entries: LibraryEntry[] }` を保持し、`path` を一意キーとする
- Rendererとは新設IPCチャンネルで通信する。内訳は次の4本
  - `library:get-all` / `library:upsert` / `library:remove` / `library:open`
- タイトル・作曲者の抽出はRendererの既存パース結果を利用し、Mainでは再パースしない
- 読み込み時のスキーマ・値域検証はフェイルソフト方式（TASK-092と同方針）

## 理由

1. **既存パターンの踏襲**: 設定はelectron-store、アノテーションはJSONサイドカーという
   既存の永続化パターンのうち、アプリ全体で共有するデータはelectron-storeが前例
2. **AppSettingsと分離する理由**: ライブラリは件数が増えるデータであり、設定の
   読み書き（起動時取得・設定変更のたびに全体をset）と混在させるとI/Oと後方互換の
   複雑さが増す。独立ストアなら`settings:set`の副作用（メニュー再構築等）とも干渉しない
3. **Main管理の理由**: ファイルパスの検証・読み取り許可リスト（PathAllowlist）登録は
   Mainの責務であり、ライブラリから開く操作をMain経由に強制することでTASK-086の
   防御を迂回させない
4. **SQLite等のDB不採用**: 想定規模（数百件程度）ではJSONで十分。ネイティブ依存の追加は
   サプライチェーン方針（Phase 18）に逆行する

## 却下した代替案

### AppSettings（recentFiles拡張）への相乗り

- 最大10件のrecentFilesを拡張する案。設定と履歴データの混在が進み、
  `settings:set`のたびにライブラリ全体が上書きされる競合面が生まれるため不採用。
  recentFilesは当面併存させ、統合の要否は別ストーリーで判断する

### JSONサイドカー（楽譜と同フォルダ）

- ライブラリは「どこに何があるか」を横断管理するデータであり、楽譜ごとの
  サイドカーでは一覧を構成できないため不採用

### Renderer側（localStorage/IndexedDB）での永続化

- ファイルパス検証・allowlist登録というMainの責務と分離してしまい、
  開く操作の安全な結線が複雑になるため不採用

## 影響

- 新設IPCチャンネル4本とpreload APIの追加（CLAUDE.mdのIPC一覧へ反映する）
- ライブラリのストアファイル（`library.json`）がuserDataフォルダに作成される
- E2Eの隔離userData方式（TASK-100）はライブラリストアにもそのまま適用できる
