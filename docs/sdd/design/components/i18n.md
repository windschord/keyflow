# I18n（UI多言語対応） コンポーネント設計

## 概要

UI表示言語の日英切り替え（US-016、REQ-016-001〜007）を実現するコンポーネント。
方式はDEC-009（依存追加なしの軽量独自i18n）に従う。

## 構成

```
src/renderer/src/lib/i18n/
├── types.ts          # Language型（'ja' | 'en'）・Messages型（typeof jaから導出）
├── ja.ts             # 日本語リソース（構造のソースオブトゥルース）
├── en.ts             # 英語リソース（Messages型への適合を型チェックで強制）
├── format.ts         # formatMessage(template, params) プレースホルダ置換の純関数
├── resolve-language.ts # resolveLanguage(stored, osLocale) 言語解決の純関数
├── index.ts          # getMessages(language) エクスポート集約
└── useTranslation.ts # useTranslation() フック（ui-sliceのlanguageを購読）

src/main/menu.ts       # メニュー用ja/enラベル辞書を内部に持つ（既存の独立保持パターン）
src/main/locale.ts     # resolveLanguage相当のMain側純関数（app.getLocale()結果を受ける）
```

## データフロー

### 起動時の言語解決（REQ-016-002/005）

```
Renderer: App.tsx 起動時
  settings:get('ui') → ui.language ('auto' | 'ja' | 'en')
  resolveLanguage(ui.language, navigator.language)
    - 'ja' / 'en' → そのまま採用
    - 'auto' または不正値 → osLocaleが'ja'始まりなら'ja'、それ以外は'en'
  → ui-slice.language へ反映

Main: createWindow() 前
  settingsService.get('ui').language と app.getLocale() で同じ規則で解決
  → createApplicationMenuTemplate({ language, ... }) でメニュー構築
```

Rendererは `navigator.language`、Mainは `app.getLocale()` を用いるが、いずれも
OSロケール由来であり同一結果になる（不一致時も次回切り替え操作で収束するため許容）。

### 言語切り替え（REQ-016-003/004）

```
SettingsModal（UIセクションの言語セレクタ「日本語 / English」）
  → ui-slice.setLanguage(lang)（Renderer UIは即時再描画）
  → settings:set('ui', { ...ui, language: lang })（永続化）
Main: settings:set ハンドラ
  → ui.language の変更を検知したら Menu.setApplicationMenu(再構築) を実行
```

新規IPCチャンネルは追加しない。既存の `settings:set` の副作用としてMainがメニューを
再構築する（結線はsrc/main/index.tsのハンドラ登録箇所）。

## 型設計

```typescript
// types.ts
export type Language = 'ja' | 'en';

// ja.ts（例。実際の構造は実装時に全UI文言を洗い出して確定する）
export const ja = {
  header: {
    openFile: 'ファイルを開く',
    play: '再生',
    // ...
  },
  settings: {
    title: '設定',
    language: '言語',
    // ...
  },
  errors: {
    parseFailed: '楽譜ファイルの読み込みに失敗しました: {message}',
    // ...
  },
} as const satisfies Record<string, Record<string, string>>;

// Messages型: jaの構造を正とする（valueはstringへ広げる）
export type Messages = DeepStringify<typeof ja>;

// en.ts
export const en: Messages = { ... }; // キー欠落はコンパイルエラー
```

- コンポーネントからは `const t = useTranslation()` で取得し `t.header.play` と参照する
- `formatMessage(t.errors.parseFailed, { message })` で `{message}` を置換する
- 翻訳キーの欠落は型エラーとなるため実行時フォールバックは原則不要だが、
  防御として `getMessages()` は `en` を基底に選択言語を上書きしたオブジェクトを返す
  （REQ-016-006の英語フォールバック）

## 状態管理

- `ui-slice` に `language: Language` と `setLanguage(language: Language)` を追加
- 初期値は `'ja'`（App.tsx起動時のresolveLanguageで直ちに上書きされる）
- `useTranslation()` は `useStore((s) => s.language)` を購読し `getMessages(language)` を返す
  （`getMessages` は言語ごとにメモ化し、毎レンダーのオブジェクト生成を避ける）

## 設定スキーマ変更

- `AppSettings.ui.language`: `string` → `'auto' | 'ja' | 'en'`（main/renderer両方の型定義）
- `DEFAULT_SETTINGS.ui.language`: `'ja'` → `'auto'`
- 既存設定ファイルとの後方互換: 保存済み `'ja'` はそのまま日本語。未知の値は
  `resolveLanguage` が `'auto'` 扱いに正規化する

## 翻訳対象（REQ-016-007）

| 領域 | 主なファイル |
|------|-------------|
| ヘッダー・操作群 | components/Header/（QuickPanel含む）、components/Toolbar/ 各コントロール |
| 設定モーダル | components/SettingsModal/index.tsx |
| Aboutモーダル | components/AboutPanel/（ライブラリ名・音源クレジット等の固有名詞は対象外） |
| コンテキストメニュー | components/NoteContextMenu/ |
| 統計・運指 | components/StatsDisplay/、components/FingeringPanel/ |
| エラー・トースト | App.tsx ほかエラー表示経路 |
| メニューバー | src/main/menu.ts（編集・表示・ウィンドウ・ヘルプ・Aboutラベル） |

対象外: 楽譜内文言（MusicXML由来データ）、`aria-label`未設定要素の新規追加は本対応の
範囲外（既存のaria-label文言は翻訳対象）。

## テスト戦略

- 純関数（`formatMessage` / `resolveLanguage` / `getMessages`のフォールバック）のユニットテスト
- `en` と `ja` のキー構造一致は型チェック（`npm run typecheck`）が担保
- ui-slice: `setLanguage` の状態遷移テスト
- SettingsModal: 言語セレクタ変更で `settings:set` が正しいペイロードで呼ばれること
- menu.ts: `language: 'en'` でラベルが英語になるテンプレート生成テスト（純関数）
- 結線テスト: Mainの `settings:set` ハンドラが `ui.language` 変更時にメニュー再構築
  コールバックを呼ぶこと（モック境界の結線テスト原則）
- E2E: 設定モーダルで言語をEnglishへ切り替え→ヘッダーの文言が英語表示に変わる→日本語へ
  戻す（ユーザー観測可能な結果を合格条件とする）

## E2Eの言語前提

既存E2Eは日本語文言のアクセシブルネームに依存している。E2E起動時は設定未保存のため
OSロケール解決となり、非日本語ロケール環境では既存E2Eが壊れる。これを避けるため、
E2Eのセットアップで `ui.language: 'ja'` を明示する（アプリ内の実UI操作または
設定ファイルのシードで実現。テスト専用の分岐をアプリ本体に入れないこと）。
