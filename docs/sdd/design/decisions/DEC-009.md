# DEC-009: UIの多言語対応を依存追加なしの軽量独自i18nで実装する

## ステータス

承認済（2026-07-12）

## 文脈

US-016でUI表示言語の日英切り替え（REQ-016-001〜007）が要件化された。
実装方式として、i18nライブラリ（react-i18next等）の導入と、TypeScriptの型システムを
活用した軽量独自実装の2案を比較する必要がある。

## 決定

**依存追加なしの軽量独自i18nを採用する。**

- 翻訳リソースはTypeScriptのネストしたオブジェクトリテラルとして定義する。
  `ja.ts` を構造のソースオブトゥルースとし、`en.ts` は `typeof ja` から導出した
  `Messages` 型への適合を型チェックで強制する
- コンポーネントは `useTranslation()` フックで現在言語のメッセージオブジェクトを取得し、
  `t.header.play` のようにプロパティアクセスで参照する（翻訳キーの欠落・タイポは
  コンパイルエラーになる）
- 可変値の埋め込みは `{name}` プレースホルダを置換する純関数 `formatMessage()` で行う
- Mainプロセス（メニューバー）は必要な文言が少数のため、既存の「main/rendererで
  型定義を独立して持つ」パターンに従い、`menu.ts` 側にメニュー用のja/enラベル辞書を持つ

## 理由

1. **依存最小化**: Phase 18でサプライチェーン強化（依存削減・Dependabot導入）を実施した
   直後であり、本番依存の追加は最小限に抑えたい。i18next + react-i18next は推移依存を含め
   相応の追加になる
2. **要件が小さい**: 対象は2言語・静的ラベル中心で、必要機能はプレースホルダ補間のみ。
   複数形処理・RTL・遅延ロード等のi18nextの主要機能は不要
3. **型安全性が高い**: プロパティアクセス方式は文字列キー方式（`t('header.play')`）と
   異なり、キーの存在・網羅性をTypeScriptの `strict: true` でコンパイル時に保証できる。
   これはREQ-016-006（欠落時フォールバック）の発生自体を型で予防する
4. **Main/Renderer分離**: react-i18nextはRenderer専用であり、メニューバー（Main側）には
   どのみち別の仕組みが必要になる

## 却下した代替案

### react-i18next + i18next

- 利点: 実績・エコシステム（抽出ツール等）・複数形/文脈対応
- 欠点: 本番依存の追加（サプライチェーン方針に逆行）、文字列キーのため型安全性の担保に
  追加の型生成の仕組みが必要、Main側は別対応が必要
- 2言語・ラベル中心の本アプリでは過剰と判断

### 翻訳リソースのJSONファイル化（src/shared/配下で共有）

- 利点: 翻訳ファイルの形式が一般的
- 欠点: JSONでは型チェック（キー網羅性）が効かない。tsconfig.node/webの
  include構成変更が必要になり、既存の「main/rendererで独立して持つ」パターンから逸脱する

## 影響

- `AppSettings.ui.language` の型を `string` から `'auto' | 'ja' | 'en'` へ厳密化する。
  既定値は `'ja'` から `'auto'` へ変更し、`'auto'` はOSロケールによる自動判定を意味する。
  既存設定ファイルに `'ja'` が保存済みの場合はそのまま日本語として尊重される
- 言語追加時は `Messages` 型に適合するリソースファイルを1つ追加し、言語ユニオン型へ
  追記するだけでよい（NFR-U-002b充足）
- 翻訳の抽出・管理ツールは使えないが、対象文言規模（数百件以下）では手動管理で足りる
